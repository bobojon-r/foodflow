## Правила написания кода

### Общие
- Весь код — асинхронный (`async def`). Никаких синхронных функций в бизнес-логике и работе с БД.
- Type hints обязательны везде.
- Docstrings и комментарии — на английском.
- Имена переменных, функций, классов — на английском, snake_case для функций/переменных, PascalCase для классов.

### FastAPI
- Все роуты версионируются: `/api/v1/...`
- Используй `APIRouter` с префиксами, не регистрируй роуты напрямую на `app`.
- Dependency Injection через `Depends()` — особенно для сессии БД, текущего пользователя, настроек.
- Pydantic-схемы в `schemas/`, SQLAlchemy-модели в `models/` — не смешивай.
- Response model указывай явно через `response_model=`.

### SQLAlchemy 2.0
- Используй новый стиль: `select(User).where(...)`, а не `session.query(User)`.
- Сессии — async: `AsyncSession`, `async with`, `await session.execute(...)`.
- `DeclarativeBase` для моделей, не устаревший `declarative_base()`.
- Типизируй колонки через `Mapped[type]` и `mapped_column(...)`.

### Pydantic v2
- `model_config = ConfigDict(...)` вместо класса `Config`.
- Для настроек — `pydantic-settings`, читай из `.env`.
- Не используй устаревшие `@validator`, используй `@field_validator`.

### Alembic
- Миграции автогенерируй: `alembic revision --autogenerate -m "description"`.
- Проверяй сгенерированные миграции перед применением — автогенерация не всегда идеальна.
- Никаких ручных изменений БД в обход миграций.

## Архитектурные принципы

- **Async-first** — вся IO асинхронная.
- **Database-per-service** — когда перейдём к микросервисам, каждый сервис владеет своей БД.
- **Event-driven между сервисами** — предпочтение Kafka, HTTP только где синхронность критична.
- **Идемпотентность** — все webhook-и и Kafka consumer-ы идемпотентны.
- **12-factor app** — конфиг через env, stateless, логи в stdout.

## Что НЕ делать

- Не предлагай Django, Flask или sync-фреймворки — проект на FastAPI.
- Не предлагай Poetry/uv/pipenv — используем `venv + requirements.txt`.
- Не добавляй технологии, которых нет в текущем этапе roadmap.
- Не используй `print()` для логирования — только `logging`.
- Не храни секреты в коде — только через env-переменные.
- Не пиши синхронные функции там, где возможен async.
- Не используй устаревший синтаксис SQLAlchemy 1.x или Pydantic v1.

## Стиль общения

- Объясняй "зачем", а не только "как". Пользователь изучает инфраструктуру, ему важен контекст.
- Если предлагаешь решение — кратко упоминай альтернативы и почему выбрал именно это.
- Большие изменения разбивай на шаги, чтобы пользователь мог идти постепенно.
- При ошибках сначала объясни причину, потом решение.
- Не перегружай формальностями, пиши по-деловому и по существу.
